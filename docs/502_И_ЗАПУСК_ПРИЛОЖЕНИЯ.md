# 502 Bad Gateway и запуск приложения

## Почему возникает 502

Ошибка **502 Bad Gateway** от nginx значит: nginx не получил ответ от бэкенда (приложение Dominion на uvicorn). Чаще всего:

1. **Процесс uvicorn не запущен** — после деплоя или перезагрузки сервера приложение не поднято.
2. **Приложение упало** — ошибка при старте (импорт, БД, порт занят).
3. **Nginx смотрит не туда** — в конфиге `proxy_pass` указан неверный адрес или порт (приложение по умолчанию слушает **8001**).

## Что проверить на сервере s-global.space

### 1. Запущено ли приложение

```bash
# Если через systemd:
sudo systemctl status dominion   # или как назван сервис

# Или поиск процесса:
ps aux | grep uvicorn
```

Если процесса нет — приложение не запущено, его нужно запустить (см. ниже).

### 2. Слушает ли порт 8001

```bash
ss -tlnp | grep 8001
# или
sudo lsof -i :8001
```

Если порт не слушается — uvicorn не работает или слушает другой порт.

### 3. Конфиг nginx

Должен быть прокси на тот же хост и порт, где работает uvicorn (обычно 127.0.0.1:8001):

```nginx
location / {
    proxy_pass http://127.0.0.1:8001;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

После правок: `sudo nginx -t && sudo systemctl reload nginx`.

### 4. Проверка живости приложения

В приложении добавлен маршрут **/health** (без авторизации). С сервера можно проверить:

```bash
curl -s http://127.0.0.1:8001/health
# Ожидается: {"status":"ok","service":"dominion"}
```

Если ответ есть — приложение живое, 502 скорее из-за nginx или сети. Если нет — приложение не отвечает или не запущено.

## Как запустить приложение

### Вариант A: вручную (для проверки)

Из каталога проекта на сервере:

```bash
cd /path/to/dominion
source venv/bin/activate
uvicorn main:app --host 0.0.0.0 --port 8001
```

Или без активации venv:

```bash
/path/to/dominion/venv/bin/uvicorn main:app --host 0.0.0.0 --port 8001
```

Для работы в фоне можно использовать `nohup` или screen/tmux.

### Вариант B: systemd (рекомендуется)

Создать файл `/etc/systemd/system/dominion.service`:

```ini
[Unit]
Description=Dominion S-GLOBAL
After=network.target postgresql.service

[Service]
Type=simple
User=www-data
Group=www-data
WorkingDirectory=/path/to/dominion
Environment="PATH=/path/to/dominion/venv/bin"
ExecStart=/path/to/dominion/venv/bin/uvicorn main:app --host 127.0.0.1 --port 8001
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

Подставить реальный `WorkingDirectory` и пути к `venv`. Затем:

```bash
sudo systemctl daemon-reload
sudo systemctl enable dominion
sudo systemctl start dominion
sudo systemctl status dominion
```

Дальше при деплое достаточно: `sudo systemctl restart dominion`.

## Краткий чеклист при 502

1. Запустить приложение (вручную или `systemctl start dominion`).
2. Проверить `curl http://127.0.0.1:8001/health`.
3. Проверить nginx: `proxy_pass` на `http://127.0.0.1:8001`, перезагрузить nginx.
4. Посмотреть логи: `journalctl -u dominion -n 100` или вывод uvicorn в консоли.
